# ТЗ для разработчика SConcur: per-coroutine context

Адресат: разработчик библиотеки `sconcur/sconcur`.
Тип: фреймворк-нейтральная фича планировщика. Никакой Laravel-специфики здесь нет и быть не
должно — она остаётся в интеграции `sconcur-laravel` (см.
[fiber-safe-laravel-bridge.ru.md](fiber-safe-laravel-bridge.ru.md)).

Версия от: 2026-06-28.

---

## 1. Цель

Дать публичный примитив **контекст текущей корутины** — key-value хранилище, привязанное к
фиберу, изолированное между конкурентными корутинами и наследуемое дочерними. Это
единственное, что нужно от библиотеки, чтобы поверх SConcur можно было построить
конкурентно-безопасную интеграцию с фреймворком (Laravel и любым другим), не свопая
глобальное состояние процесса.

Аналог в других рантаймах: `Async\current_context()` в PHP TrueAsync,
`context.Context`/goroutine-local в Go, coroutine context в Swoole/Hyperf.

---

## 2. Зачем (контекст задачи)

SConcur HTTP-сервер исполняет каждый запрос в отдельном `Fiber` конкурентно в одном процессе
(`HttpServer` → `Scheduler::serve` → `Scheduler::spawn` на запрос). Когда хендлер уходит в
async-вызов и засыпает, планировщик пускает другую корутину. Любая интеграция с фреймворком,
у которого есть per-request состояние (текущий запрос, пользователь, сессия, локаль), не может
хранить это состояние в общих синглтонах/статиках — оно перетрётся соседней корутиной.

Решение, не требующее свопа глобалок: хранить per-request состояние в контексте корутины и
читать его по `Fiber::getCurrent()`. Библиотека уже ведёт per-fiber реестр в `SConcur\State`,
поэтому естественно расширить именно его.

---

## 3. Что НЕ требуется (важно, чтобы не переусложнить)

- **Не** свопать `Container::setInstance` / фасады / любое глобальное состояние — это забота
  интегратора, не библиотеки.
- **Не** добавлять хуки «activate/deactivate вокруг resume» — в модели с контекстом они не
  нужны: код читает контекст по текущему фиберу в момент обращения.
- **Не** тащить в библиотеку зависимость на Laravel/PSR/фреймворк. Контекст хранит
  произвольные `mixed` значения по строковым ключам.

---

## 4. Требуемый публичный API

```php
namespace SConcur\Scheduler; // или SConcur\Context — на усмотрение

interface Context
{
    /** Значение по ключу или null, если нет (с учётом наследования от родителя). */
    public function find(string $key): mixed;

    /** true, если ключ виден (свой или унаследованный). */
    public function has(string $key): bool;

    /**
     * Записать значение в КОНТЕКСТ ТЕКУЩЕЙ корутины (локально, не у родителя).
     * $replace=false и ключ уже есть локально — не перезаписывать (no-op);
     * $replace=true — перезаписать.
     */
    public function set(string $key, mixed $value, bool $replace = false): void;

    /** Удалить локальный ключ (унаследованный от родителя не трогает). */
    public function forget(string $key): void;
}

/** Контекст текущей корутины (по Fiber::getCurrent()); вне фибера — корневой контекст. */
function current_context(): Context;
```

`current_context()` — основная точка входа. Допускается также метод фасада
`Scheduler::get()->currentContext()` — на вкус API библиотеки, но свободная функция удобнее
для интегратора.

## 5. Семантика

### 5.1. Привязка к корутине

Контекст идентифицируется текущим фибером: `spl_object_id(Fiber::getCurrent())` — тот же
ключ, по которому `State::getCurrentFlow()` уже резолвит flow. У каждой spawned-корутины и
каждой корутины WaitGroup — свой контекст.

### 5.2. Доступ вне фибера

`current_context()`, вызванный вне любого фибера (главный поток, бутстрап, между тиками),
обязан вернуть **корневой контекст** процесса, а не бросать исключение и не возвращать null.
Это нужно, чтобы код интегратора, обращающийся к контексту на этапе инициализации (до первого
запроса), не падал.

### 5.3. Наследование дочерними корутинами

Когда корутина порождает дочернюю, дочерняя должна **видеть** ключи родителя. Точки рождения:

- `Scheduler::spawn()` — groupless (один HTTP-запрос);
- `WaitGroup::add()` — вложенные корутины (например, параллельные Mongo-запросы внутри одного
  HTTP-запроса). Без наследования такие дочерние корутины не увидят request/пользователя
  родителя.

Рекомендуемая модель — **read-through по цепочке родителей** (а не копирование):

- `find()/has()` ищут ключ в собственной карте корутины, при отсутствии — поднимаются к
  родителю, и так до корня.
- `set()` пишет **локально** в корутину (родителя не мутирует) — дочерняя может «затенить»
  ключ родителя, не повредив его и соседние корутины.

Привязка «родитель» фиксируется в момент создания фибера: родитель = корутина, в которой
выполняется `spawn()`/`WaitGroup::add()` (текущий фибер на этот момент).

Опционально: параметр изоляции на создание корутины (например
`spawn(Closure $cb, bool $inheritContext = true)`), чтобы можно было создать корутину с
чистым контекстом без родителя. По умолчанию — наследовать.

> Альтернатива read-through — отдать дочерней ту же ссылку на `Context` родителя (shared).
> Проще, но запись в дочерней протекает в родителя и в соседние дочерние. Для per-request
> данных, которые в основном читаются, это часто допустимо, но read-through безопаснее и
> рекомендуется по умолчанию.

### 5.4. Жизненный цикл и очистка

Контекст корутины создаётся лениво (при первом `set`, либо при `registerFiberFlow`) и
**уничтожается вместе с корутиной**. Освобождение обязано происходить там же, где библиотека
уже снимает per-fiber учёт:

- `State::unRegisterFiber($fiberId)` — снять и контекст этого фибера;
- `State::deleteFlow($flowKey)` / `Scheduler::forget()` / `detach()` / `completeCoroutine()` /
  `failCoroutine()` — финальная очистка.

Требование: после завершения N запросов в памяти не должно оставаться N контекстов (нет
утечки ссылок на значения — иначе интегратор будет течь объёмом per-request объектов).

### 5.5. Реентерабельность через suspend/resume

Контекст обязан переживать произвольное число suspend/resume одной корутины: он привязан к
стабильному id фибера, поэтому значение, записанное до `await`, доступно после `resume` той же
корутины. Никаких действий на самих `start()/resume()` для этого не требуется.

### 5.6. Конкурентность и производительность

- Среда кооперативная, однопоточная (фиберы не вытесняются произвольно), поэтому блокировки не
  нужны.
- `find/set/has` — горячий путь (зовутся на каждый scoped-резолв интегратора). Доступ должен
  быть O(1) по собственной карте + подъём по цепочке родителей (глубина обычно 1–2: запрос →
  вложенная корутина).

## 6. Точки интеграции в текущем коде

| Что | Файл / место | Действие |
|---|---|---|
| Хранилище контекстов | `src/State.php` (рядом с `$fiberFlows`) | добавить `array<int $fiberId, контекст>` + ссылку на родителя |
| Резолв текущего | `src/State.php::getCurrentFlow()` (логика по `Fiber::getCurrent()`) | по тому же принципу добавить `currentContext()` |
| Инициализация + наследование | `Scheduler::spawn()` (`new Fiber` … `registerFiberFlow` … `$fiber->start()`, стр. ~85–111) | при создании фибера запомнить родителя = текущий фибер |
| Инициализация + наследование | `WaitGroup::add()` (`new Fiber` стр.61 … `registerFiberFlow` стр.65 … `start()` стр.87) | то же для вложенных корутин |
| Очистка | `State::unRegisterFiber()` / `State::deleteFlow()` | снять контекст фибера |
| Публичная функция | новый файл (напр. `src/Context/functions.php` с `current_context()`) | автозагрузка через composer `files` |

`Scheduler::resumeCoroutine()` (стр.~344) трогать **не нужно** — контекст резолвится по
текущему фиберу в момент обращения, а не по событию resume.

## 7. Скетч реализации (ориентир, не обязателен буквально)

```php
// src/State.php (добавления)
/** @var array<int, array<string, mixed>> собственные значения контекста по fiberId */
protected static array $fiberContext = [];
/** @var array<int, int> fiberId дочерней → fiberId родителя */
protected static array $fiberParent = [];

public static function registerCoroutineContext(int $fiberId, ?int $parentFiberId): void
{
    static::$fiberContext[$fiberId] = [];
    if ($parentFiberId !== null) {
        static::$fiberParent[$fiberId] = $parentFiberId;
    }
}

public static function ctxFind(int $fiberId, string $key): mixed
{
    $cur = $fiberId;
    while ($cur !== null) {
        if (array_key_exists($key, static::$fiberContext[$cur] ?? [])) {
            return static::$fiberContext[$cur][$key];
        }
        $cur = static::$fiberParent[$cur] ?? null; // подъём к родителю
    }
    return null;
}

public static function ctxSet(int $fiberId, string $key, mixed $value, bool $replace): void
{
    if (!$replace && array_key_exists($key, static::$fiberContext[$fiberId] ?? [])) {
        return;
    }
    static::$fiberContext[$fiberId][$key] = $value;
}

// в unRegisterFiber(): unset(static::$fiberContext[$fiberId], static::$fiberParent[$fiberId]);
```

`current_context()` — тонкая обёртка: берёт `spl_object_id(Fiber::getCurrent())` (или
корневой псевдо-id вне фибера) и проксирует в `State::ctx*`.

`родитель` для `registerCoroutineContext` вычисляется в `spawn()`/`WaitGroup::add()` как
`Fiber::getCurrent()` на момент создания дочернего фибера.

## 8. Крайние случаи

1. **Вне фибера** — корневой контекст (см. §5.2), не исключение.
2. **Сирота/детач** (`detach`, orphan по `masterPid`) — контекст снимается вместе с фибером.
3. **Глубокая вложенность** — цепочка родителей; убедиться, что при завершении дочерней её
   звено снимается, а контекст родителя остаётся, пока жив родитель.
4. **Завершение родителя раньше дочерней** — для HTTP-flow обычно не бывает (родитель ждёт
   группу), но реализация не должна падать, если родительское звено уже снято: подъём по
   цепочке просто прерывается.
5. **Большие/тяжёлые значения** — библиотека их не интерпретирует; ответственность за объём на
   интеграторе, но §5.4 (очистка) обязана гарантировать освобождение по завершении корутины.

## 9. Критерии приёмки (фреймворк-нейтральные тесты)

1. **Изоляция.** Две одновременно живые spawned-корутины: `set('k', 'A')` в первой и
   `set('k', 'B')` во второй; каждая читает своё значение, не чужое.
2. **Переживание await.** Корутина: `set('k','v')` → async-пауза (`Sleeper`) → после resume
   `find('k') === 'v'`, при том что между паузой и resume отработала другая корутина.
3. **Наследование.** Родитель `set('req', X)` → `WaitGroup::add` дочерней; дочерняя
   `find('req') === X`.
4. **Затенение без протечки.** Дочерняя `set('req', Y)`; дочерняя видит `Y`, родитель
   по-прежнему видит `X`; соседняя дочерняя видит `X`.
5. **Очистка.** После завершения K корутин `State` не хранит их контексты (проверка размера
   внутренних массивов / отсутствие роста под нагрузкой).
6. **Вне фибера.** `current_context()->set/find` в главном потоке работают на корневом
   контексте без исключений.

---

## 10. Объём и приоритет

- Минимально достаточный объём — §4 + §5.1, §5.2, §5.4, §5.5 (контекст на корутину + корневой
  + очистка + переживание resume). Этого хватает для spawn-on-request HTTP.
- Наследование §5.3 нужно, как только хендлер начнёт порождать вложенные корутины (параллельные
  Mongo внутри запроса). Желательно сразу, чтобы не переделывать.
- Параметр изоляции (§5.3, опционально) — по желанию, не блокирует.

После появления этого примитива со стороны `sconcur-laravel` убирается весь Octane-своп
глобалок (`clone $app`, `CurrentApplication::set`, `flush`), и per-request состояние уезжает в
`current_context()`. Подробности интеграции — в
[fiber-safe-laravel-bridge.ru.md](fiber-safe-laravel-bridge.ru.md).
